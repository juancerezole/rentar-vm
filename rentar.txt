┌────────────────────────────────────────────────────────────────────────────┐
│ 1. RESUMEN EJECUTIVO                                                       │
└────────────────────────────────────────────────────────────────────────────┘

Rentar es un portal de alquileres hyperlocal para Villa María, Córdoba.
La idea es convertirse en "la página de la ciudad" para alquileres, con
foco en datos de mercado locales y herramientas que las inmobiliarias chicas
no tienen disponibles en Zonaprop/ArgenProp.

Estado actual: DEMO funcional para presentar a inmobiliarias y generar
interés/leads. NO es producción. Hay decisiones técnicas tomadas conscientemente
que no escalan (SQLite, sin auth fuerte, etc.).

Estimación de mercado: 30-50 inmobiliarias activas en Villa María; ~85k habs.
Proyección realista año 2: USD$15-35k/año netos para una persona sola.
Proyección año 3+: USD$40-80k/año con expansión a 1-2 ciudades vecinas.

Modelo de negocio (acordado, NO implementado):
  - Tier 1: banners locales (USD$70-230/mes c/u, 4-6 anunciantes)
  - Tier 2: publicaciones destacadas (USD$3-8 por propiedad/mes)
  - Tier 3: suscripción Premium para inmobiliarias (USD$25-50/mes)
  - Tier 4: directorio de profesionales matriculados (USD$80-150/año)

│ 2. STACK TECNOLÓGICO                                                       │
└────────────────────────────────────────────────────────────────────────────┘

BACKEND
  - Node.js 22.5+ (usa node:sqlite incorporado, NO better-sqlite3)
  - Express 4
  - SQLite vía node:sqlite (DatabaseSync)
  - JWT (jsonwebtoken)
  - bcryptjs para hashing de passwords
  - Sin ORM (queries SQL directas)

FRONTEND
  - React 18 + Vite 5
  - React Router 6
  - TailwindCSS 3 (con paleta custom; emerald remapeado a verde oscuro)
  - Axios para HTTP
  - lucide-react para iconos
  - Fuentes: Sora (cuerpo) + DM Serif Display (titulares)

NO HAY:
  - ORM
  - tests
  - CI/CD
  - Sentry / error tracking
  - object storage (las fotos son URLs externas a Unsplash)
  - WebSockets / realtime

│ 4. SCHEMA DE BASE DE DATOS                                                 │
└────────────────────────────────────────────────────────────────────────────┘

users
  id, nombre, email (unique), password_hash, rol, empresa, telefono, created_at
  rol ∈ {admin, inmobiliaria, usuario}

barrios
  id, nombre (unique), x, y, precio_mes_anterior
  x/y son coordenadas relativas (% del SVG del mapa) — NO son lat/long

properties
  id, user_id (FK→users), titulo, tipo, direccion, barrio,
  precio, precio_anterior (NULL si no es liquidación),
  ambientes, banos, superficie,
  garantia (sin/requerida/ambas), mascotas, amoblado, expensas_incluidas,
  destacado, liquidacion, descripcion, imagen, created_at
  tipos: departamento, casa, duplex, campo, galpon, comercial,
         cochera, baulera, contenedor

banners
  id, titulo, subtitulo, color, link, activo

profesionales
  id, nombre, rol, matricula, rating, iniciales, color, telefono, email

NO hay índices definidos. Hay que agregar (ver sección 9).

  - 10 barrios reales de Villa María

│ 5. ENDPOINTS API                                                           │
└────────────────────────────────────────────────────────────────────────────┘

PÚBLICOS
  GET  /api/health                        ping
  GET  /api/properties                    listado con filtros
                                          query: tipo, barrio, minPrecio, maxPrecio,
                                          ambientes, garantia, mascotas, amoblado,
                                          expensas, q, destacado, liquidacion
  GET  /api/properties/:id                detalle
  GET  /api/properties/cheapest-by-barrio más económica por barrio
  POST /api/properties/ai-search          búsqueda en lenguaje natural
                                          body: { text }
                                          OJO: es regex matching, no IA real
  GET  /api/barrios                       listado con cantidad por barrio
  GET  /api/barrios/heatmap               heatmap con avg/min/max + delta vs mes ant.
  GET  /api/banners                       banners activos
  GET  /api/profesionales                 listado de profesionales
  GET  /api/stats/summary                 dólar/clima/propiedades/inmobiliarias
                                          DÓLAR Y CLIMA HARDCODEADOS — ver punto crítico

AUTENTICACIÓN
  POST /api/auth/register                 body: { nombre, email, password, rol, empresa, telefono }
                                          rol solo puede ser usuario|inmobiliaria
  POST /api/auth/login                    body: { email, password } → { token, user }
  GET  /api/auth/me                       requiere Authorization: Bearer <token>

INMOBILIARIA (rol: inmobiliaria | admin)
  GET    /api/properties/mine/list        propiedades del usuario logueado
  POST   /api/properties                  crear propiedad
  PUT    /api/properties/:id              editar (dueño o admin)
  DELETE /api/properties/:id              eliminar (dueño o admin)

ADMIN (rol: admin)
  GET    /api/admin/users                 listado completo de usuarios
  PUT    /api/admin/users/:id/rol         cambiar rol
  DELETE /api/admin/users/:id             eliminar usuario (CASCADE → propiedades)

REQUISITOS
  - Node.js 22.5+ (usa node:sqlite, módulo experimental en 22-23, estable cerca de 24)

    Si "npm" da error de ExecutionPolicy:

INSTALACIÓN (primera vez)
  cd server
  npm install
  cd ../client
  npm install

ARRANCAR EN DESARROLLO (2 terminales)

  # Terminal 1 - Backend (puerto 4000)
  cd server
  npm run dev

  # Terminal 2 - Frontend (puerto 5173)
  cd client
  npm run dev

  Abrir http://localhost:5173

RESETEAR DB (si los datos se rompen o cambia el schema)
  cd server
  npm run reset
  npm run dev

  Esto borra database.sqlite y regenera el seed completo.

BUILD DE PRODUCCIÓN (todo en uno)
  npm install   <- en root, instala client+server (vía script "build" del root)
  npm run build <- en root
  npm start     <- en root, levanta server que sirve el frontend buildeado

│ 8. PLAN DE DEPLOY                                                          │
└────────────────────────────────────────────────────────────────────────────┘

ELECCIÓN: Railway + dominio en NIC.ar

  - Backend + frontend juntos en Railway (un solo servicio).
  - Express sirve el dist/ del cliente cuando NODE_ENV=production.
  - Costo: USD$5/mes Railway Hobby + ~USD$10/año dominio = USD$70/año.

PASOS (cuando se decida deployar)

  1. git init + push a un repo privado en GitHub
  2. Comprar dominio rentar.com.ar en NIC.ar (~AR$10.000/año)
  3. Conectar Railway al repo
  4. Build Command: npm run build
  5. Start Command: npm start
  6. Env vars en Railway:
       NODE_ENV=production
       JWT_SECRET=<un random largo>
       PORT=4000
  7. Generar dominio Railway, después conectar el dominio propio (CNAME)

DECISIÓN PENDIENTE: agregar Volume persistente o no.
  - Sin Volume: la DB se resetea en cada deploy. BUENO para demo (siempre limpia).
  - Con Volume: datos persisten. Necesario apenas haya inmobiliarias reales cargando.

NO USAR Render free tier: tiene cold starts de 30s, mata cualquier reunión de venta.


┌────────────────────────────────────────────────────────────────────────────┐
│ 9. PROBLEMAS CRÍTICOS PARA PRODUCCIÓN (priorizado)                         │
└────────────────────────────────────────────────────────────────────────────┘

Esta lista es OBLIGATORIA antes de cualquier cliente que pague.
Estimación: ~2 semanas FT o 1 mes part-time.

PRIORIDAD 1 — bloqueantes absolutos
  [ ] Migrar SQLite → PostgreSQL.
      SQLite serializa escrituras y NO sincroniza WAL entre instancias.
      Con 100 inmobiliarias escribiendo es corrupción silenciosa.
      Recomendado: Supabase o Neon (free tier alcanza al inicio).
      Reescribir queries SQL (la mayoría es portable directo).

  [ ] Cerrar CORS. Hoy en prod usa origin: true (refleja CUALQUIER origen).
      Cualquier sitio puede hacer requests autenticados.
      Cambiar a whitelist explícita: ['https://rentar.com.ar'].
      Archivo: server/index.js línea ~17

  [ ] Rate limiting en /auth/login.
      Usar express-rate-limit. 5 intentos cada 15 min por IP+email.
      Sin esto, fuerza bruta trivial sobre todas las cuentas.

  [ ] JWT_SECRET obligatorio en startup.
      Hoy hace fallback a 'dev-secret'. Si la env var falta en prod,
      cualquiera firma JWTs como admin.
      Cambiar a: if (!process.env.JWT_SECRET) throw new Error(...)
      Archivo: server/middleware/auth.js línea ~3

  [ ] Validación de inmobiliarias.
      Hoy cualquiera se registra como rol="inmobiliaria" y publica al toque.
      Sin verificación de matrícula CUCICBA o equivalente provincial.
      Solución: registro → estado pendiente → admin verifica → puede publicar.

  [ ] Password reset flow.
      Si una inmobiliaria olvida password, NO HAY recovery.
      Mail con token de 1 hora vía Resend o SES.

  [ ] Implementar botón "Solicitar más información" en PropertyDetail.
      Hoy no hace nada. Es EL botón de conversión.
      Sin lead capture funcional el modelo de negocio no funciona.

PRIORIDAD 2 — alto riesgo a corto plazo
  [ ] Paginación en GET /api/properties.
      Hoy devuelve TODO. Con 20k propiedades es 5MB+ y el browser muere.
      LIMIT 24 OFFSET ? + ?page=N

  [ ] Índices en propiedades.
      CREATE INDEX idx_props_barrio ON properties(barrio);
      CREATE INDEX idx_props_tipo ON properties(tipo);
      CREATE INDEX idx_props_precio ON properties(precio);
      CREATE INDEX idx_props_user ON properties(user_id);
      Ídem users(email).

  [ ] Validación de inputs con Zod o similar.
      precio acepta negativos. titulo/descripcion sin max length.
      email sin regex. password sin requisitos. imagen acepta cualquier string.
      1 día de laburo, 100 incidentes evitados.

  [ ] Upload real de imágenes (object storage).
      Hoy las inmobiliarias pegan URLs de Unsplash. Ningún cliente real
      va a hacer eso. Necesario:
        - Cloudflare R2 o S3
        - POST /api/upload con multer
        - sharp para resize/compresión
        - Tabla property_images (property_id, url, orden)
        - Galería con varias fotos por propiedad
      ~3 días.

  [ ] Reemplazar LIKE %% por full-text search.
      Cuando se migre a Postgres, usar to_tsvector + índice GIN.
      O Meilisearch/Typesense para mejor UX.

  [ ] Sacar el badge "IA" o conectar un LLM real.
      Hoy es regex matching, no IA. "Buscador inteligente con IA"
      es publicidad engañosa. Opciones:
        a) Usar Claude Haiku (USD$0.25/MTok input, casi gratis) para
           extraer filtros estructurados del texto.
        b) Renombrar a "Búsqueda en lenguaje natural" sin "IA".

  [ ] Audit log.
      Tabla audit_log (user_id, action, entity_type, entity_id,
                       before_json, after_json, ip, ts).
      Middleware que logea cada mutation.

  [ ] Prevención de "último admin se demote".
      Hoy admin puede demoteare a usuario y nadie puede entrar al admin nunca más.
      Validar en /admin/users/:id/rol.

  [ ] Soft delete de usuarios.
      Hoy DELETE es CASCADE permanente.
      Agregar deleted_at, recovery 30 días.

  [ ] Implementar el corazón ❤️ de favoritos o sacarlo.
      Hoy decorativo, parece roto.
      Requiere tabla favoritos + auth.

PRIORIDAD 3 — operacional / compliance
  [ ] Sentry para error tracking (free tier 5k eventos/mes).

  [ ] Logs estructurados (pino + request-id middleware).

  [ ] Smoke tests mínimos (5-10 endpoints más usados).

  [ ] Compliance Ley 25.326 (datos personales).
      OBLIGATORIO POR LEY ARGENTINA:
        - Política de privacidad (link en footer)
        - Términos y condiciones
        - Endpoint para que el usuario pida eliminación de sus datos
        - Registro de la base de datos en la AAIP
      Multas hasta AR$5M. Abogado especializado: USD$300 una vez.

  [ ] SEO / SSR.
      Hoy es SPA, Google indexa mal.
      Migrar a Next.js (mismo React, mismos componentes) con SSR + meta
      tags por propiedad. ~2-3 días. Es CRÍTICO para el modelo de negocio
      porque los usuarios llegan vía Google.

  [ ] Analytics para inmobiliarias.
      Sin "tu propiedad tuvo X visitas" no se justifica que paguen.
      Plausible o Umami (~USD$9/mes) + tabla property_views.

  [ ] Filtros y paginación en el dashboard de inmobiliaria.
      Una inmobiliaria con 80 propiedades se vuelve loca.

  [ ] Datos de dólar/clima REALES.
      Hoy hardcodeado: $1180/$1210, 22°C nublado. Si un cliente nota la
      mentira, pierde confianza en TODA la plataforma.
      Bluelytics y OpenWeather: ambos free, 1 hora de trabajo.

  [ ] Accesibilidad: aria-labels, contraste, navegación por teclado.

  [ ] UptimeRobot para monitoreo (free tier 50 checks).

PRIORIDAD 4 — deuda técnica
  [ ] Graceful shutdown (SIGTERM cierra DB y conexiones).
  [ ] app.set('trust proxy', 1) detrás de Railway.
  [ ] Helmet para security headers.
  [ ] Timeout en axios (frontend cuelga si backend no responde).
  [ ] Refresh tokens para JWT (hoy mueren a 7 días sin warning).
  [ ] Mobile-first refactor. Hoy sidebar de filtros es desastre en móvil.
  [ ] CI/CD básico (GitHub Actions: lint + build).
  [ ] Random seed en banners (hoy siempre el mismo orden).
  [ ] Timing attack en login: ejecutar bcrypt.compare con hash dummy
      cuando el email no existe, para igualar tiempos.
  [ ] Tokens en httpOnly cookie en lugar de localStorage (XSS robaría todo).

┌────────────────────────────────────────────────────────────────────────────┐
│ 12. GOTCHAS / TRAMPAS CONOCIDAS                                            │
└────────────────────────────────────────────────────────────────────────────┘

DB MIGRATION (ya implementada)
  - server/db.js tiene función migrate() que detecta:
      a) si falta la columna barrios.precio_mes_anterior → la agrega
      b) si hay emails @alquilavm.com legacy → los actualiza a @rentar.com.ar
  - Cada seed (users, barrios, properties, banners, profesionales) corre
    INDEPENDIENTEMENTE si su tabla está vacía.
  - Si querés regenerar todo: npm run reset (en server/)

NODE:SQLITE
  - Es el SQLite incorporado en Node 22.5+. Reemplaza a better-sqlite3.
  - El módulo está marcado experimental → emite ExperimentalWarning.
  - Por eso los scripts usan --no-warnings.
  - Si en algún momento Node hace breaking changes a node:sqlite, hay que migrar.

LASTINSERTROWID ES BIGINT
  - En node:sqlite, info.lastInsertRowid devuelve BigInt (no Number).
  - Hay que convertir con Number(...) antes de pasarlo a otra query
    o de comparar con números.

POWERSHELL EXECUTION POLICY
  - En Windows fresh, npm.ps1 está bloqueado por default.
  - Solución: Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

ONE DRIVE + NODE_MODULES
  - El proyecto está en OneDrive. OneDrive intenta sincronizar node_modules.
  - Si pasa: pausar OneDrive antes de npm install.
  - Mejor: mover el proyecto fuera de OneDrive a futuro (ej C:\dev\rentar).

PROXY DE VITE
  - vite.config.js redirige /api → http://localhost:4000.
  - Si cambia el puerto del backend, actualizar ahí también.
